## 認証と認可

似ているようで違う2つの概念です。

| 概念 | 英語 | 意味 | 例 |
|-----|------|------|-----|
| **認証** | Authentication | あなたは誰？ | ログイン |
| **認可** | Authorization | 何ができる？ | 権限チェック |

### 認証（Authentication）

「あなたは誰ですか？」を確認すること。

- ログイン画面でメールとパスワードを入力
- サーバーがユーザーを特定
- 「田中さんですね」と確認

### 認可（Authorization）

「あなたは何ができますか？」を確認すること。

- 田中さんは管理者？一般ユーザー？
- 管理者なら大会を削除できる
- 一般ユーザーなら閲覧のみ

## なぜ認証が必要？

「Reactで作る！フロントエンド完結アプリ開発」のPart 3で作ったアプリは、誰でも全ての操作ができました。

本番環境では、以下が必要です。

- **誰が操作したかを記録**したい
- **自分のデータだけ**を編集させたい
- **管理者だけ**に削除を許可したい

これらを実現するのが認証・認可です。

## セッション認証 vs トークン認証

認証には主に2つの方式があります。

### セッション認証

```
1. ログイン
   ブラウザ ──────→ サーバー
          「ID: tanaka, PW: 1234」

2. セッション作成
   ブラウザ ←────── サーバー
          「セッションID: abc123」
          （Cookieに保存）

3. 以降のリクエスト
   ブラウザ ──────→ サーバー
          「Cookie: abc123」
          サーバーがセッションIDで
          ユーザーを特定
```

**特徴:**
- サーバーがセッション情報を保持
- Cookie でセッションIDを送受信
- サーバーの負荷が高い

### トークン認証（JWT）

```
1. ログイン
   ブラウザ ──────→ サーバー
          「ID: tanaka, PW: 1234」

2. トークン発行
   ブラウザ ←────── サーバー
          「トークン: eyJhbGci...」
          （localStorageに保存）

3. 以降のリクエスト
   ブラウザ ──────→ サーバー
          「Authorization: Bearer eyJhbGci...」
          サーバーがトークンを検証
```

**特徴:**
- サーバーはセッション情報を保持しない
- トークン自体にユーザー情報が含まれる
- サーバーの負荷が低い（スケールしやすい）

## JWT（JSON Web Token）

トークン認証でよく使われる形式です。

### JWTの構造

```
eyJhbGciOiJIUzI1NiJ9.eyJ1c2VySWQiOiIxIn0.xxxx
└──────────────────┘ └──────────────────┘ └──┘
      ヘッダー              ペイロード      署名
```

| 部分 | 内容 |
|-----|------|
| ヘッダー | アルゴリズムなどのメタ情報 |
| ペイロード | ユーザーIDなどのデータ |
| 署名 | 改ざん検知用のハッシュ |

### ペイロードの例

```json
{
  "userId": "1",
  "email": "tanaka@example.com",
  "role": "admin",
  "exp": 1704067200  // 有効期限
}
```

:::message alert
**注意**: ペイロードはBase64エンコードされているだけで、**暗号化されていません**。
誰でも中身を見られるので、パスワードなどの機密情報は入れないでください。
:::

### JWTの検証

1. トークンを受け取る
2. 署名を検証（改ざんされていないか）
3. 有効期限を確認
4. ペイロードからユーザー情報を取得

署名の検証には**秘密鍵**が必要なので、サーバーだけが正しく検証できます。

## 認証の流れ

```
┌──────────────────────────────────────────────────────┐
│ 1. サインアップ                                        │
│    ユーザー登録（メール、パスワード）                    │
│    パスワードはハッシュ化して保存                       │
├──────────────────────────────────────────────────────┤
│ 2. ログイン                                           │
│    メール・パスワードで認証                             │
│    JWTトークンを発行                                   │
├──────────────────────────────────────────────────────┤
│ 3. 認証が必要なリクエスト                              │
│    トークンをヘッダーに付けてリクエスト                  │
│    サーバーがトークンを検証                             │
├──────────────────────────────────────────────────────┤
│ 4. ログアウト                                         │
│    クライアント側でトークンを削除                       │
└──────────────────────────────────────────────────────┘
```

## パスワードのハッシュ化

パスワードは**絶対に**そのまま保存してはいけません。

```
パスワード: password123
    ↓ ハッシュ化
ハッシュ値: $2b$10$abcdef...（元に戻せない）
```

ハッシュ化すると、元のパスワードを復元できません。
ログイン時は、入力されたパスワードをハッシュ化して比較します。

## この章のまとめ

- 認証は「誰か」を確認、認可は「何ができるか」を確認
- トークン認証（JWT）はサーバーの負荷が低くスケールしやすい
- JWTはヘッダー、ペイロード、署名の3部分で構成
- パスワードは必ずハッシュ化して保存
