## なぜNext.jsでバックエンドを作るのか？

バックエンドを作る言語・フレームワークは多数あります。Python（Django, FastAPI）、Ruby（Rails）、Go、Java（Spring）など、選択肢は豊富です。

この本では**Next.js**を使います。Next.jsは**Node.js**（JavaScriptをサーバー側で実行する環境）上で動作するフレームワークです。その理由は：

### 1. 新しい言語を学ばなくていい

フロントエンドで使った**JavaScript/TypeScript がそのまま使えます**。新しい言語の文法を覚える必要がありません。

### 2. 同じプロジェクト内でバックエンドが書ける

Next.js の **API Routes** 機能を使えば、フロントエンドと同じプロジェクト内にバックエンドのコードを書けます。別のサーバーを立てる必要がありません。

```
my-app/
├── app/
│   ├── page.tsx          # フロントエンド
│   └── api/
│       └── users/
│           └── route.ts  # バックエンド（API）
```

### 3. デプロイが簡単

Vercel を使えば、フロントエンドもバックエンドも**ワンクリックでデプロイ**できます。サーバーの設定やインフラ管理は不要です。

### 4. フルスタック開発の入門に最適

フロントエンドとバックエンドを一貫して学べるので、Web開発の全体像を理解しやすくなります。

:::message
**他の言語を学ぶ必要がないわけではありません**

実務では、以下のような場合に Go や Python などの他の言語を検討することもあります：

**他の言語を検討すべきケース**
- **パフォーマンスが極めて重要**: 大量のリクエストを処理する必要がある（Go、Rust）
- **既存システムとの連携**: 企業の既存システムが特定の言語で書かれている（Java、.NET）
- **機械学習・データ分析**: AI機能を組み込みたい（Python）
- **マイクロサービス化**: アプリが大規模になり、サービスごとに最適な言語を選びたい
- **チームのスキルセット**: チームに特定の言語の専門家がいる

ただ、**最初の一歩**としては、すでに知っている JavaScript/TypeScript で始めるのが効率的です。アプリが拡大してから、必要に応じて他の技術を学ぶのが現実的なアプローチです。
:::
